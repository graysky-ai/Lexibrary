# Library Structure

This document describes the anatomy of the `.lexibrary/` directory -- what each artifact type is, where it lives, and how it is produced.

## Directory Layout

After running `lexictl init` and `lexictl update`, your `.lexibrary/` directory looks like this:

```
.lexibrary/
├── config.yaml              # Project configuration
├── START_HERE.md            # Agent entry point -- project topology and navigation
├── src/                     # Mirror tree of design files (matches your source tree)
│   └── mypackage/
│       ├── .aindex          # Directory routing table
│       ├── main.py.md       # Design file for main.py
│       ├── utils.py.md      # Design file for utils.py
│       └── models/
│           ├── .aindex      # Directory routing table for models/
│           └── user.py.md   # Design file for user.py
├── concepts/                # Concepts wiki -- project vocabulary and patterns
│   ├── pydantic-config.md
│   └── change-detection.md
├── stack/                   # Stack Q&A knowledge base
│   ├── ST-001-fix-import-cycle.md
│   └── ST-002-api-timeout-handling.md
├── index.db                 # Link graph SQLite index (gitignored)
├── daemon.pid               # Daemon PID file (present only when daemon is running)
└── logs/                    # Daemon log files
```

## Artifact Types

### config.yaml

The project configuration file. Created by `lexictl init` and edited manually or via `lexictl setup --update`. Contains all settings that control Lexibrary's behavior: scope root, LLM provider, token budgets, ignore patterns, daemon settings, and more.

See [Configuration](configuration.md) for the full reference.

**Produced by:** `lexictl init` (created), manual editing (updated).

### START_HERE.md

The entry point for AI agents. Regenerated automatically during every `lexictl update`. Contains:

- **Project topology** -- High-level directory structure with descriptions.
- **Package map** -- What each major directory contains.
- **Navigation guidance** -- How to find specific files and concepts.

Agents are instructed to read this file at the start of every session. It gives them a high-level orientation before they dive into specific files.

**Produced by:** `lexictl update` (regenerated every run).

### Design Files (mirror tree)

Design files are Markdown files that describe individual source files. They live in a mirror tree inside `.lexibrary/` that matches your source directory structure:

```
Source file:      src/mypackage/utils.py
Design file:      .lexibrary/src/mypackage/utils.py.md
```

Each design file contains:

- **YAML frontmatter** -- Agent-editable metadata:
  - `description` -- A short summary of what the file does.
  - `updated_by` -- Either `"archivist"` (generated by Lexibrary) or `"agent"` (manually edited by an agent).

- **Staleness metadata** -- An HTML comment footer with hashes and timestamps used for change detection:
  - `source` -- Relative path to the source file.
  - `source_hash` -- SHA-256 hash of the source file at generation time.
  - `interface_hash` -- Hash of the public interface skeleton (if AST extraction is enabled).
  - `design_hash` -- Hash of the design file content.
  - `generated` -- Timestamp of when the design file was generated.
  - `generator` -- Identifier of the generation tool (e.g., `lexibrary-v2`).

- **Summary** -- What the file does and why it exists.
- **Key details** -- Important implementation notes.
- **Interface skeleton** -- For supported languages (Python, TypeScript, JavaScript), a tree-sitter-extracted skeleton of public functions, classes, and exports.
- **Wikilinks** -- References to related concepts using `[[concept-name]]` syntax.

**Produced by:** `lexictl update` (generated and regenerated via LLM).

**Consumed by:** `lexi lookup <file>` (agents read these before editing source files).

### .aindex Files (routing tables)

`.aindex` files are per-directory routing tables that provide a quick index of what a directory contains. Each `.aindex` file lives inside the corresponding directory in the `.lexibrary/` mirror tree.

An `.aindex` file contains:

- **Billboard** -- A one-line description of the directory's purpose.
- **Entries** -- A list of files and subdirectories with their types and descriptions.
- **Local conventions** -- Coding conventions that apply to files in this directory (e.g., "all modules use `from __future__ import annotations`").
- **Staleness metadata** -- Same format as design files, for change detection.

Agents can read `.aindex` files to understand a directory at a glance without reading every individual design file. Conventions from `.aindex` files are inherited by child directories and surfaced by `lexi lookup`.

**Produced by:** `lexictl update` (refreshed during project updates), `lexictl index <dir>` (generated on demand).

**Updated by:** `lexi describe <dir> <description>` (agents can update the billboard).

### concepts/ Directory

The concepts wiki is a collection of Markdown files in `.lexibrary/concepts/`. Each concept defines a piece of project-specific vocabulary, an architectural pattern, or a domain term.

A concept file contains:

- **YAML frontmatter:**
  - `title` -- The concept name.
  - `aliases` -- Alternative names for the concept (for wikilink resolution).
  - `tags` -- Categorization tags.
  - `status` -- One of `draft`, `active`, or `deprecated`.
  - `superseded_by` -- (Optional) Name of the concept that replaces this one if deprecated.

- **Body** -- Free-form Markdown describing the concept.

Concepts are referenced from design files using wikilink syntax: `[[concept-name]]`. The validator checks that all wikilinks resolve to existing concepts.

**Produced by:** `lexi concept new <name>` (created from template), manual editing (updated).

**Consumed by:** `lexi concepts [topic]` (search/list), `lexi concept link <concept> <file>` (link to design files).

### stack/ Directory

The Stack Q&A knowledge base lives in `.lexibrary/stack/`. Each post is a Markdown file named with an auto-assigned ID and a slug derived from the title (e.g., `ST-001-fix-import-cycle.md`).

A Stack post contains:

- **YAML frontmatter:**
  - `id` -- Auto-assigned identifier (e.g., `ST-001`).
  - `title` -- The post title.
  - `tags` -- Categorization tags (at least one required).
  - `status` -- One of `open`, `resolved`, `outdated`, or `duplicate`.
  - `created` -- Creation date.
  - `author` -- Who created the post.
  - `bead` -- (Optional) Associated bead ID.
  - `votes` -- Vote count.
  - `duplicate_of` -- (Optional) ID of the original post if this is a duplicate.
  - `refs` -- Cross-references to other artifacts:
    - `concepts` -- List of concept names.
    - `files` -- List of source file paths.
    - `designs` -- List of design file paths.

- **Problem section** -- Description of the issue.
- **Evidence section** -- Supporting evidence or context.
- **Findings** -- Solutions contributed by agents or operators, each with:
  - `number` -- Finding number (F1, F2, ...).
  - `date` -- When the finding was posted.
  - `author` -- Who wrote the finding.
  - `body` -- The finding text.
  - `votes` -- Vote count for this finding.
  - `accepted` -- Whether this finding has been accepted.
  - `comments` -- Discussion comments on the finding.

**Produced by:** `lexi stack post --title --tag` (created from template), manual editing (updated).

**Consumed by:** `lexi stack search`, `lexi stack view`, `lexi stack list` (search and read), `lexi stack finding`, `lexi stack vote`, `lexi stack accept` (contribute and curate).

### index.db (Link Graph)

A SQLite database that indexes relationships between all artifacts. It is gitignored because it can be rebuilt from the artifact files. The link graph contains 8 tables:

| Table | Purpose |
|-------|---------|
| `meta` | Schema version and metadata |
| `artifacts` | Registry of all indexed artifacts (design files, concepts, Stack posts) |
| `links` | Directed edges between artifacts (imports, wikilinks, file refs, concept refs) |
| `tags` | Tag assignments on artifacts |
| `aliases` | Concept aliases |
| `conventions` | Convention entries from `.aindex` files |
| `build_log` | Build history for tracking when the index was last updated |
| `artifacts_fts` | FTS5 full-text search index over artifact content |

The link graph accelerates several operations:

- **Reverse dependency lookups** -- `lexi lookup` uses the link graph to show which files import a given file.
- **Tag-based search** -- `lexi search --tag` queries the link graph for tag matches.
- **Full-text search** -- `lexi search <query>` uses the FTS5 index when available.
- **Validation** -- Several validator checks (bidirectional dependencies, dangling links, orphan artifacts) query the link graph.

When the link graph is not available (e.g., `index.db` was deleted), commands gracefully degrade -- they still work but without index-accelerated queries.

**Produced by:** `lexictl update` (built automatically after design file generation).

**Rebuilt by:** Delete `index.db` and run `lexictl update` to rebuild from scratch.

### daemon.pid

A file containing the process ID of the running daemon. Present only when `lexictl daemon start` or `lexictl sweep --watch` is active. If the daemon crashes, a stale PID file may remain -- `lexictl daemon status` detects and cleans up stale PID files.

**Produced by:** `lexictl daemon start`, `lexictl sweep --watch`.

### logs/ Directory

Log files produced by the daemon when running in background mode. The log level is controlled by the `daemon.log_level` config setting (default: `info`).

**Produced by:** The daemon process during sweep and watchdog operations.

## What Gets Gitignored

The following items inside `.lexibrary/` should typically be gitignored:

- `index.db` -- Rebuilt from artifacts, can be large.
- `daemon.pid` -- Transient process file.
- `logs/` -- Operational log files.

Everything else (config, design files, `.aindex` files, concepts, Stack posts, `START_HERE.md`) should be committed to version control. These artifacts represent accumulated project knowledge and should be shared with the team.
