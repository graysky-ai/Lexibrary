## Context

Conventions are referenced in 8+ locations across the Lexibrary codebase — model field, generator, serializer, parser, link graph schema/builder/query, CLI lookup, validator, and BAML prompts — but the entire pipeline produces zero data. The generator hard-codes `local_conventions=[]`, so every downstream consumer is a no-op. The only conventions agents can find are hallucinated by the START_HERE LLM prompt from directory structure alone.

The conventions-artifact plan (D1-D12) settles all design decisions. This change implements file-based conventions as a first-class artifact, mirroring the concepts system, with CLI commands for creation and lifecycle management.

## Goals / Non-Goals

**Goals:**
- Establish `.lexibrary/conventions/*.md` as the source of truth for project conventions
- Provide `ConventionIndex` for scope-aware convention retrieval (project-wide and directory-scoped)
- Enable convention creation via `lexi convention new` (agents and users)
- Deliver conventions through `lexi lookup` with display limit and priority ordering
- Support draft/active/deprecated lifecycle with agent sign-off workflow
- Index conventions into the link graph for FTS search and graph queries
- Allow user-declared conventions in config as a seeding mechanism

**Non-Goals:**
- LLM-extracted conventions (archivist pipeline) — deferred to v2
- Pattern-based scopes (glob/fnmatch matching) — deferred to v2
- Auto sign-off via LLM-as-judge — deferred to v2
- Convention conflict detection in validator — deferred to v2
- Migration tooling — pre-launch, no existing data to migrate (D8)

## Decisions

### File-based storage, parallel to concepts (D1)

Each convention is a standalone `.lexibrary/conventions/<slug>.md` file with YAML frontmatter and markdown body. This mirrors `.lexibrary/concepts/*.md`.

**Alternatives considered:**
- Database-only (link graph `conventions` table): Poor editability (no text editor access), poor versioning (SQLite binary not diffable). Rejected.
- Hybrid (conventions in `.aindex` + link graph): `.aindex` files are regenerated by the indexer — conventions would be overwritten. The `list[str]` type lacks metadata. Rejected.

**Why file-based wins:** Best editability (human-readable markdown), best versioning (git-diffable), consistent with concepts pattern, and the downstream link graph becomes a derived index rather than source of truth.

### Clean cut: remove `local_conventions` from `.aindex` (D9)

The `local_conventions` field, serializer section, parser block, and CLI walk are all removed. No deprecation period because the field was never populated and we're pre-launch (D8).

### ConventionIndex as single retrieval path (D10)

`ConventionIndex` loaded from `.lexibrary/conventions/` is the primary retrieval path. The link graph `conventions` table is a derived index for FTS and graph queries only. This mirrors how `ConceptIndex` works.

### Slug-based file naming (D11)

Convention files use `kebab-case-slug.md` derived from the title (lowercase, replace spaces/special chars with hyphens, truncate at 60 chars, numeric suffix on collision). Concepts use PascalCase; conventions use slugs because convention titles are sentences/phrases, not noun phrases.

### Two-part body format (D12)

First paragraph is the prescriptive rule (shown in `lexi lookup`). Everything after is optional rationale/context (shown in `lexi conventions <name>`). The parser extracts a `rule` property from the first paragraph.

### Scope resolution: directory + project only (D7)

v1 supports `scope: project` (applies everywhere) and `scope: <directory-path>` (applies within subtree). The `scope` field is a string, forward-compatible with glob patterns in v2. Resolution builds an ancestry chain from the target file to scope root, collects matching conventions, orders root-to-leaf, and applies display limit (default 5) favoring leaf-ward specificity.

### Display limit with priority ordering (D5)

Default limit of 5 conventions per `lexi lookup`. Configurable via `conventions.lookup_display_limit`. When truncating, keep most-specific (leaf-ward) conventions, drop most-general (root-ward). Within same scope, order by priority descending then title alphabetically.

### Config-declared conventions materialized as files

Conventions declared in `config.yaml` under a `conventions` key are materialized into `.lexibrary/conventions/` during `lexictl update` (or a new `lexictl convention sync` command). They get `source: config`, `status: active`, and are authoritative. This is a seeding mechanism — once materialized, the files are the source of truth.

## Module structure

```
src/lexibrary/conventions/
├── __init__.py          # Re-exports ConventionIndex, ConventionFile
├── index.py             # ConventionIndex class (load, find_by_scope, search, filter)
├── parser.py            # parse_convention_file() — YAML frontmatter + markdown body
└── serializer.py        # serialize_convention_file() — write convention to disk

src/lexibrary/artifacts/
└── convention.py        # ConventionFile, ConventionFileFrontmatter Pydantic models
```

## Risks / Trade-offs

- **[File proliferation]** → Projects with many directory-scoped conventions may create dozens of files. Mitigated by slug naming, display limit, and the fact that conventions are typically terse.
- **[Stale conventions]** → Conventions may reference directories that no longer exist. Mitigated by future validator checks (v2 scope). For v1, conventions with invalid scopes still load but won't match any files.
- **[Agent noise]** → Agents may create low-quality conventions. Mitigated by draft/approve workflow — agent-created conventions start as `status: draft` and require human approval.
- **[Display budget]** → 5-convention limit may hide important conventions. Mitigated by priority field (user conventions default to 0, agent conventions to -1) and truncation notice pointing to `lexi conventions <path>`.
- **[Config duplication]** → Config-declared conventions exist in both `config.yaml` and as files. The files are the source of truth once materialized; config is a seeding mechanism. A `lexi convention sync` command handles the materialization.
