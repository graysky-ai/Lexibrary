## Context

START_HERE.md is currently generated by an LLM via the BAML function `ArchivistGenerateStartHere`. It produces 5 sections (topology, ontology, navigation_by_intent, navigation_protocol, convention_index) assembled into markdown by `_assemble_start_here()` in `start_here.py`. The archivist pipeline calls `generate_start_here()` as Step 5 after all file-level design files are processed.

The problem: 4 of 5 sections are redundant or hallucinated. Topology duplicates what `.aindex` billboard summaries already provide. Ontology duplicates `lexi concepts`. Navigation protocol is in 6+ other places. Convention index has no backing data source.

Rather than incrementally removing sections, we delete the entire LLM generation pipeline and replace it with a single procedural artifact: `.lexibrary/TOPOLOGY.md`, built directly from `.aindex` data.

## Goals / Non-Goals

**Goals:**
- Remove all BAML/LLM infrastructure for START_HERE generation
- Create a new `topology.py` module with procedural topology builder
- Output `.lexibrary/TOPOLOGY.md` with adaptive-depth annotated tree
- Maintain backward compatibility — next `lexi update` replaces old START_HERE with new TOPOLOGY
- Mark superseded plans

**Non-Goals:**
- Navigation-by-intent enrichment (blocked on upstream, Phase 4)
- Conventions as first-class artifacts (blocked on upstream, Phase 5)
- Hook-based context injection (separate concern, extracted to future plan)
- Cleaning up old `.lexibrary/START_HERE.md` files in user projects (they'll just be stale)

## Decisions

### D1: Delete entire LLM pipeline, not incremental removal

**Choice**: Delete `start_here.py`, BAML function, service types in one change rather than phased section-by-section removal.

**Rationale**: The user's direction is clear — get rid of the LLM-generated artifact entirely. Phased removal adds complexity for no benefit when the end state is full deletion. The only section worth keeping (navigation-by-intent) is blocked on richer inputs anyway.

**Alternative considered**: Phased removal (plan Phases 1→2→3). Rejected because it creates intermediate states that still require LLM calls for diminishing value.

### D2: New module `topology.py` rather than gutting `start_here.py`

**Choice**: Create `src/lexibrary/archivist/topology.py`. Delete `src/lexibrary/archivist/start_here.py`.

**Rationale**: The module name should reflect its purpose. `start_here.py` implies a multi-section document; `topology.py` is accurate. Clean separation also makes it trivial to add back a richer START_HERE later if needed.

### D3: Output as `TOPOLOGY.md` not `START_HERE.md`

**Choice**: Write to `.lexibrary/TOPOLOGY.md`.

**Rationale**: The file is no longer a "start here" guide — it's a structural map. The name should match the content. Agents and docs will reference `TOPOLOGY.md`.

### D4: Adaptive depth based on project scale

**Choice**: Three tiers based on directory count from `.aindex` data:
- Small (≤10 dirs): full tree, no filtering
- Medium (11–40 dirs): depth ≤2 + hotspots (dirs with >5 child entries)
- Large (41+ dirs): depth ≤1 + hotspots + top-N by child count

**Rationale**: Small projects should show everything. Large projects need editorial filtering or the topology becomes noise. The `.aindex` `entries` list gives us child counts for hotspot detection without extra crawling.

### D5: Graceful fallback when no `.aindex` files exist

**Choice**: Return informative placeholder text rather than raising.

**Rationale**: A freshly-initialized project won't have `.aindex` files until after its first `lexi update` processes files. The topology generator runs early in the pipeline, so it needs to handle this gracefully.

### D6: Remove `start_here_tokens` config field

**Choice**: Remove the `start_here_tokens` field from `TokenBudgetConfig` since there's no LLM output to budget.

**Rationale**: Procedural output is deterministic — token budgeting is meaningless. The field becomes dead config. Clean removal avoids confusion.

**Alternative considered**: Repurpose as `topology_max_lines`. Rejected — the adaptive depth algorithm already handles output size.

## Risks / Trade-offs

**[Old START_HERE.md files linger]** → Users who don't run `lexi update` after upgrading will still have the old `START_HERE.md`. Mitigation: docs mention this; `lexi update` is the standard workflow. We do NOT auto-delete `START_HERE.md` — that could destroy hand-edited files in edge cases.

**[Navigation-by-intent loss]** → Removing START_HERE entirely means losing the navigation-by-intent table until Phase 4 unblocks. Mitigation: this section's quality was limited anyway (LLM only sees billboard summaries). The topology tree with billboard annotations provides directional guidance in the interim.

**[Agents referencing START_HERE.md]** → Agent rules (CLAUDE.md, orientation.md) currently tell agents to read START_HERE.md. Mitigation: update all references to TOPOLOGY.md as part of this change.

**[No rollback of user data]** → Once `lexi update` writes TOPOLOGY.md, the old START_HERE.md content is gone (it was overwritten each run anyway). Mitigation: this is existing behavior — START_HERE was always regenerated, never preserved.
