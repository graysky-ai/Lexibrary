KEY FINDINGS FROM SEARCH COMMAND ANALYSIS
==========================================================

1. COMMAND LANDSCAPE (5 commands, not 1)
   - lexi search              → Unified cross-artifact search
   - lexi concepts            → Concept vocabulary discovery
   - lexi conventions         → Project rules & coding practices
   - lexi stack search        → Issue/solution discovery
   - lexi stack list          → Issue status browsing (minimal overlap)

2. CRITICAL UNIQUENESS (Cannot consolidate without loss)

   A. Conventions Dual-Mode Query (ONLY in lexi conventions)
      - Single positional arg: path OR free-text
      - Heuristic: contains "/" or "." → path; else → free-text
      - Path mode: finds conventions applicable to directory (with inheritance)
      - Example: "lexi conventions src/auth" finds scoped conventions for that dir
      - FRAGILE: "Docker.io" or "config.auth.verify" misidentified as paths

   B. Conventions Path-Scoped Inheritance (ONLY in lexi conventions)
      - "lexi conventions src/auth" finds conventions where:
        - scope == "project" (global), OR
        - scope == "src/auth" or any parent directory
      - lexi search --scope uses simple prefix matching (different semantics)
      - Different output: inheritance ≠ prefix matching

   C. Stack Specialized Filters (ONLY in lexi stack search)
      - --concept <name>: filter posts referencing a specific concept
      - --resolution-type <type>: filter by fix/workaround/etc.
      - No equivalent in other commands
      - Support multi-filter intersection: query + tag + concept + status all AND-combined

   D. Concepts AND-Tag Logic (UNIQUE to lexi concepts)
      - --tag python --tag security means BOTH tags required
      - lexi search --tag uses OR logic (single tag across all types)
      - lexi conventions also uses AND, but different from concepts

   E. Concepts Exclude Deprecated by Default (UNIQUE behavior)
      - --all flag to include deprecated
      - Different from stack (which shows all by default)

4. CONSOLIDATION VERDICT

   Could lexi search replace all? NO
   - Would lose path-scoped inheritance
   - Would lose AND-tag filtering per artifact type
   - Would lose Stack-specific filters (concept, resolution_type)
   - Would increase CLI complexity significantly

   Could stack list merge to stack search? MOSTLY
   - Functional overlap: "stack list --status open" ≈ "stack search --status open"
   - UX distinction: "list" → fast listing; "search" → discovery with query
   - Current semantic clarity has value (minor consolidation candidate)

5. ARCHITECTURE NOTES

   Current design philosophy:
   - SPECIALIZED > UNIFIED: each command optimized for workflow
   - FALLBACK > REQUIRED: link graph optional, file-scanning fallback works
   - EXPLICITNESS > INFERENCE: users explicitly control scope

   Link graph acceleration:
   - Tag searches: O(1) via index instead of file scan
   - Free-text: FTS5 relevance ranking instead of substring
   - Graceful fallback when link graph missing

6. IMPLEMENTATION FRAGILITY ALERTS

   A. Conventions dual-mode heuristic is weak
      ```python
      if query and ("/" in query or "." in query):
          # Assume it's a path
      ```
      Risk: Queries like "Docker.io", "config.auth.verify" misidentified as paths
      Fix: Check if path exists before treating as path, or explicit --path flag

   B. Conventions path-scoped inheritance not reflected in link graph
      - CLI walks parent dirs from file location (works)
      - Link graph has synthetic conventions rows but not used (design intact, implementation missing)
      - Two parallel data access paths
