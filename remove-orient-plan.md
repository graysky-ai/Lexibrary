# Plan: Remove `orient` Command and `lexi-orient` Skill

## Context

The `lexi orient` command aggregates topology, library stats, and IWH signals into a single output for agent session start. Its responsibilities are already covered by reading `TOPOLOGY.md` directly and `lexi iwh list` for signals. The command is being fully removed -- not deprecated -- to simplify the CLI surface and agent workflow.

**Key finding:** The `core_rules.md` template (canonical agent rules shipped to all environments) already does NOT reference `lexi orient` -- it says "read `.lexibrary/TOPOLOGY.md` at session start". Only the orient-specific skill, CLI command, init/rules infrastructure, agent templates, docs, and tests need removal.

**Replacement workflow for agent templates:** "Read `.lexibrary/TOPOLOGY.md` for project layout, then run `lexi iwh list` to check for pending signals."

## Risks

1. **Test volume** (~300+ lines across 11 files) -- mitigated by merging tests into Phase 2 and running per-file.
2. **Agent template coherence** -- 3 templates + 3 dogfood copies must have identical replacement workflow.
3. **Init rules generation** -- Phases 2.3-2.6 modify rule generators that ship to OTHER projects. Integration tests must pass.
4. **TOPOLOGY.md** is generated -- manual edits are stopgap. User should run `lexictl update` after all code changes.
5. **Stale orient skill cleanup** is out of scope -- older projects with `.claude/skills/lexi-orient/` will need to remove it manually. No new cleanup code will be added to `generate_claude_rules()`.
6. **Existing user configs** -- Projects with `orientation_tokens` in their `config.yaml` will not break (`ConfigDict(extra="ignore")` silently drops unknown fields). No migration action needed.

## Follow-up: Improve Config Validation Error Messages

**Out of scope for this plan, but recommended.**

`load_config()` in `src/lexibrary/config/loader.py` currently lets `pydantic.ValidationError` propagate raw (line 135). When a config file has invalid values, users see a Pydantic traceback with no indication of which file or YAML line is at fault.

**Recommended improvement:** Catch `ValidationError` in `load_config`, enrich with the source file path (global vs project) and the offending field path/value, and re-raise with a user-friendly message. This would help with any future config field changes, not just orient removal.
