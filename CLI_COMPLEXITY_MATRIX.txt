================================================================================
COMPLEXITY DEFINITIONS
================================================================================

SIMPLE:
  - 1-2 required arguments
  - 0-2 optional flags
  - Minimal validation logic
  - Single operation (no conditionals)
  - Examples: orient, help, concept new

MODERATE:
  - 1-3 optional arguments or 3-5 optional flags
  - Filter chains (AND/OR logic)
  - Optional-with-dependency validation
  - Multiple code paths (e.g., scope vs. query mode)
  - Examples: concepts, stack search, conventions

COMPLEX:
  - 6+ options
  - Conditional validation (e.g., --resolve requires --finding)
  - Multiple internal operations (lookup does 6+ distinct searches)
  - Priority-based truncation or traversal algorithms
  - One-shot workflows that combine multiple steps
  - Examples: lookup, stack post, impact

================================================================================
COMPLEXITY DRIVERS (What Makes Commands Complex)
================================================================================

1. ARGUMENT/OPTION COUNT
   - lookup: 1 arg (complex internally) → COMPLEX
   - concepts: 4 options + 1 optional arg → MODERATE
   - orient: 0 args, 0 options → SIMPLE

2. VALIDATION COMPLEXITY
   - Stack post: --resolve requires --finding, --resolution-type requires --resolve
     → Conditional validation (COMPLEX)
   - Conventions: dual-mode argument (path vs. query) → MODERATE
   - Concept new: straightforward template → SIMPLE

3. INTERNAL OPERATIONS
   - lookup: 6+ distinct searches (design, conventions, issues, dependents, etc.)
     + token budget truncation → COMPLEX
   - stack search: single search + multiple filters → MODERATE
   - orient: single output generation → SIMPLE

4. FILTERING/MATCHING LOGIC
   - concepts: --tag filter uses AND logic → MODERATE
   - conventions: dual-mode query interpretation → MODERATE
   - Stack commands: filter chains → MODERATE
   - Simple commands: no filtering → SIMPLE

5. STATEFUL BEHAVIOR / IDEMPOTENCY
   - IWH write: overwrites (idempotent) → SIMPLE
   - Stack finding: appends (not idempotent) → MODERATE
   - Concept deprecate: already-deprecated check → SIMPLE

6. OUTPUT FORMAT OPTIONS
   - Most commands: Rich tables only → SIMPLE
   - lookup: multiple output types (markdown + tables) → adds complexity
   - impact: has --quiet mode (paths only) → adds option overhead

OVERALL VERDICT:

  Main gaps: structured output formats and workflow documentation.

REQUIRED VS. OPTIONAL ARGUMENT PATTERNS
================================================================================

PATTERN A: All positional (1-2 required args)
  - lookup <file>
  - impact <file>
  - concept new <name>
  - concept link <name> <file>
  - stack view <id>
  - stack finding <id>
  - stack vote <id> up|down
  Usage: agent remembers argument order; minimal option confusion

PATTERN B: All options (no positionals)
  - convention new --scope ... --body ... [--tag ...] [--title ...] [--source ...] [--alias ...]
  Usage: Avoids ambiguity; harder to remember order; good for complex workflows

PATTERN C: Mixed (1 positional + options)
  - concepts [topic] --tag ... --status ... --all
  - conventions [query] --tag ... --status ... --scope ... --all
  - search [query] --tag ... --scope ...
  Usage: Natural for agents; optional positional comfortable

PATTERN D: Flags only (no args or options)
  - orient
  - help
  - iwh list
  Usage: Simplest; used for listings and outputs

RECOMMENDATION: Patterns C and D dominate (good). Pattern B (all options) used
sparingly (convention new). Pattern A mixed with required args (mostly simple).

OUTPUT FORMATS ACROSS COMMANDS
================================================================================

Rich Tables (most common):
  - concepts, conventions, stack search, stack list
  Pros: colored, aligned, readable
  Cons: no machine parsing; no JSON alternative

Markdown + Tables (lookup):
  - Design file content + sections + link tables
  Pros: readable, structured by sections
  Cons: large output, truncation silent

Markdown Panels (stack view):
  - Formatted header + problem + findings + comments
  Pros: visually organized
  Cons: not machine-parseable

Tree ASCII (impact):
  - Hierarchical tree with descriptions + warnings
  Pros: clear dependency chains
  Cons: only one format (has --quiet for paths)

Plain Text (orient):
  - Topology.md + file descriptions + stats + IWH summaries
  Pros: lightweight, integrates with LLM context
  Cons: no structure; token budget truncation silent

Confirmation Text (mutations):
  - "Created X", "Updated Y"
  Pros: simple, adequate
  Cons: no structured feedback

GAP: No --format json across any command. Plans exist but not implemented.

================================================================================
STATEFUL VS. STATELESS ANALYSIS
================================================================================

All 31 commands are STATELESS:
  - No session object or in-memory state
  - Each command reads/writes files independently
  - Multiple runs don't interfere (except for non-idempotent mutations)

Idempotent Commands (Safe to Repeat):
  ✓ concept link        — "already linked" → noop
  ✓ concept deprecate   — "already deprecated" → noop
  ✓ convention approve  — "already active" → noop
  ✓ iwh write           — overwrites existing signal → deterministic

Non-Idempotent Commands (Accumulate):
  ✗ stack post          — each run creates new post with new ID
  ✗ stack finding       — each run appends new finding (F1, F2, ...)
  ✗ stack vote          — each run increments vote count
  ✗ stack comment       — each run appends new comment
  ✗ concept comment     — each run appends new comment
  ✗ convention comment  — each run appends new comment
  ✗ design comment      — each run appends new comment

IMPACT: Agents can safely repeat idempotent commands but must be careful with
mutations (no transactional grouping). Good design overall.

RECOMMENDED AGENT COMMAND SEQUENCES (from CLAUDE.md)
================================================================================

SESSION START:
  1. lexi orient              # Project context + stats
  2. lexi iwh list            # Check for previous-session signals
  3. lexi iwh read <dir>      # Consume signals

BEFORE EDITING A FILE:
  1. lexi lookup <file>       # Design file + conventions + links

BEFORE ARCHITECTURAL DECISIONS:
  1. lexi concepts <topic>    # Concept wiki search

WHEN DEBUGGING:
  1. lexi stack search <query> # Find related issues + attempts

AFTER SOLVING AN ISSUE (One-Shot):
  lexi stack post \
    --title "Title" \
    --tag category \
    --problem "Description" \
    --finding "Solution" \
    --resolve \
    --resolution-type fix

OR (Multi-Step):
  1. lexi stack post --title ... --tag ... --problem ...
  2. lexi stack finding <id> --body ...
  3. lexi stack accept <id> --finding 1 --resolution-type fix

LEAVING WORK INCOMPLETE:
  lexi iwh write <dir> --scope incomplete --body "..."
  lexi iwh write <dir> --scope blocked --body "..."
