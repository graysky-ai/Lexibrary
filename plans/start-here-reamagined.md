## Problems

### 1. Ontology overlaps with Concepts (and is inferior)

The ontology section asks an LLM to invent 5–15 domain term definitions from directory names and billboard text. Meanwhile, the Concepts wiki (`lexi concepts`) exists specifically to be the authoritative domain glossary — with full definitions, aliases, lifecycle status, and cross-references.

**Result**: The LLM guesses at ontology from structural hints while a proper glossary exists elsewhere. The two can (and do) drift apart. An agent that reads START_HERE's ontology may get a shallower or conflicting understanding compared to the concept files.

**Verdict**: Ontology should not be LLM-generated from directory trees. It should either be procedurally assembled from the top concept files, or replaced with a pointer to `lexi concepts`.

### 2. Convention Index is the only conventions source — and it's hallucinated

This is the most critical problem. The `.aindex` `local_conventions` field is **hard-coded to `[]`** in `indexer/generator.py:166`. The link graph's `conventions` table derives from `.aindex` and is therefore also empty. `lexi lookup`'s "Applicable Conventions" section has nothing to show.

START_HERE's `convention_index` is currently the **only place conventions exist in the entire system** — and it's generated by an LLM guessing from directory names. The conventions in Lexibrary's own hand-maintained `blueprints/START_HERE.md` (e.g., `from __future__ import annotations`, `pathspec: "gitignore"`) are accurate only because a human wrote them.

For other projects, the LLM has to infer conventions from structural signals alone. It cannot read source code (only billboard summaries). This is unreliable.

**Verdict**: Conventions need a real source of truth. Either:
- Populate `local_conventions` from the archivist pipeline (LLM reads actual source code during design file generation and extracts conventions)
- Or let users declare conventions in config, which flow into `.aindex` files

START_HERE should then assemble conventions procedurally from `.aindex`, not hallucinate them.

### 3. Navigation Protocol is redundant with agent rules

The "how to use .lexibrary" bullets are already injected into every agent environment:
- `base.py` core_rules: "read START_HERE", "run `lexi lookup` before editing", "update design file after"
- `claude.py`: hooks mechanically enforce lookup-before-edit and design-file-update reminders
- `cursor.py`: `lexibrary-editing.mdc` scoped conditional rule
- `orientation.md`, `quick-reference.md`: both restate the protocol

Navigation protocol appears in **6+ places**. Including it in START_HERE means the LLM regenerates something that should be stable text, risking subtle wording drift from the canonical rules.

**Verdict**: Navigation protocol should live in agent rules (where it's enforced) and docs (where it's explained). START_HERE should not include it — or should include it as a static injected block, not LLM-generated.

### 4. Topology is useful but its purpose is poorly defined and its generation is wasteful

Topology's core purpose is to **give agents a high-level overview of how the project is laid out and how to navigate it**. That purpose is currently lost in the implementation — the LLM re-synthesises the directory tree that was already fed as input, trimming it to 10–20 key lines and adding annotations. But the `.aindex` Child Map tables already provide per-directory annotated listings, and the raw tree is available from the filesystem.

The value-add is *editorial*: deciding which directories/files matter most. An LLM does this reasonably but expensively (a full-project LLM call for 10–20 lines of tree).

There is also significant overlap between topology and the **root-level `.aindex` file**, which already provides a billboard summary and a Child Map of top-level entries with descriptions. For small projects, the root `.aindex` *is* essentially the topology. For larger projects, the topology needs to go deeper — but how deep should be adaptive, not fixed.

**Verdict**: Topology should be procedurally generated from `.aindex` entries, with the depth/breadth of the tree adapting to project scale. See [Procedural Generation: Topology](#topology--adaptive-procedural-generation) for the proposed algorithm.

### 5. Navigation by Intent is genuinely valuable — but could be better sourced

The task-to-file routing table is the most unique and useful section. No other artifact provides this. The link graph *could* answer "which files relate to X?" but requires tool access; START_HERE puts the shortcuts directly in context.

However, the LLM generates this table from billboard summaries — it has no knowledge of actual agent workflows, common tasks, or which files are most frequently edited. The table quality depends entirely on how good the billboards are.

**Verdict**: This is the strongest candidate for keeping LLM-generated, but it should be enriched with better inputs (design file descriptions, concept tags, Stack post topics — not just billboards).

### 6. The generation pipeline is expensive and coarse

`generate_start_here()` is called on every `lexictl update` (full project update only). It makes one LLM call with the entire directory tree and all billboard summaries. There is no incremental update — if one file changes, the whole thing regenerates. And since `update --changed-only` skips START_HERE entirely, it drifts after incremental updates until the next full update.

**Verdict**: The all-or-nothing approach doesn't scale. Procedurally-generated sections wouldn't need LLM calls at all. LLM-generated sections (if any remain) should only regenerate when their inputs actually change.

## What Should Change

### Proposed Structure: Split, Source, and Inject

The core insight is: **START_HERE tries to be five things at once, generated by one LLM call from weak inputs**. Each section has a better source of truth or delivery mechanism.

| Current Section | Proposed Disposition | New Source | Delivery |
|-----------------|---------------------|------------|----------|
| **Topology** | Keep, but procedurally generate | `.aindex` billboard summaries + directory tree depth/size heuristics | Procedural — no LLM |
| **Ontology** | Remove from START_HERE | Concept files (`lexi concepts`) | Pointer in START_HERE: "Run `lexi concepts` for domain glossary" |
| **Navigation by Intent** | Keep, LLM-generated with richer inputs | Design file frontmatter descriptions + concept tags + link graph edges | LLM call (the only one) |
| **Convention Index** | Remove from START_HERE; fix real source | `.lexibrary/conventions/` file-based artifacts (decided — see `conventions-artifact.md` D1) | Procedural assembly from convention files; surfaced via `lexi lookup` |
| **Navigation Protocol** | Remove from START_HERE | Agent rules (`base.py` core_rules) | Injected at `lexictl init` / `lexictl setup`; lives in CLAUDE.md / .cursor/rules / AGENTS.md |

### What goes where

**In START_HERE.md** (slimmed down, mostly procedural):
1. **Project Topology** — procedurally generated annotated tree (billboard summaries as annotations, depth-limited, largest/most-complex dirs first)
2. **Navigation by Intent** — the only LLM-generated section; enriched with design file descriptions and concept cross-references
3. A **pointers block** (static, injected):
   - "Domain vocabulary: run `lexi concepts` or browse `.lexibrary/concepts/`"
   - "Conventions: run `lexi lookup <file>` to see applicable conventions"
   - "Workflow: see your agent rules file (CLAUDE.md / .cursor/rules / AGENTS.md)"

**In agent rules** (CLAUDE.md / .cursor/rules / AGENTS.md):
- Navigation protocol (already there via `base.py` core_rules)
- No change needed — this is already correct

**In `.lexibrary/conventions/` convention files** (needs implementation):
- File-based convention artifacts, mirroring the concepts structure
- Per-directory and pattern-based scopes for targeted rules
- Surfaced to agents via `lexi lookup`'s "Applicable Conventions" output

**In Concept files** (already there):
- Domain ontology — no change needed, just stop duplicating it in START_HERE

## Procedural Generation Opportunities

### Topology — adaptive procedural generation

The key insight is that topology should **scale with the project**. A 5-directory project doesn't need the same treatment as a 200-directory monorepo. The algorithm should adapt based on the relationship between node count (directories) and tree depth.

**Proposed algorithm:**

```
1. Collect all .aindex files in the mirror tree
2. Build the directory graph:
   - nodes = number of directories with .aindex files
   - max_depth = deepest .aindex relative to scope_root
   - breadth_ratio = nodes / max_depth  (wide vs deep indicator)

3. Calculate display depth:
   - Small project (nodes ≤ 10): show everything — full tree
   - Medium project (nodes 11–40): show depth ≤ 2, plus any deeper
     directory with >5 child entries (hotspots)
   - Large project (nodes 41+): show depth ≤ 1, plus hotspots,
     plus directories with highest child entry counts (top N)

4. For each displayed directory:
   - Render as indented tree line
   - Annotate with billboard summary (from .aindex)
   - If truncated (children not shown), append "(N subdirs)"

5. Root .aindex integration:
   - The root .aindex billboard becomes the topology header/summary line
   - Root .aindex Child Map entries provide the top-level skeleton
   - Deeper .aindex files extend the tree where needed
```

**Overlap with root `.aindex`:** The root `.aindex` Child Map and the topology's top level are the same data. Rather than duplicating, topology should *start from* the root `.aindex` and extend downward. For very small projects, topology = root `.aindex` annotated tree, and the section might be omitted entirely in favour of a pointer: "See root `.aindex` for project layout."

No LLM needed. The quality depends on billboard quality, which is already LLM-generated during design file updates.

### Ontology — pointer only (for now)

Replace with a static pointer: "Run `lexi concepts` or browse `.lexibrary/concepts/` for domain vocabulary."

A future enhancement could procedurally surface the top N concepts ranked by popularity or connectivity. This would require a ranking signal — e.g., a voting/usage system for concepts, or link graph in-degree counts (how many design files, Stack posts, and other concepts reference a given concept). Neither exists today.

**Decision**: Start with a pointer. Monitor whether agents effectively discover concepts on their own. If not, revisit with a procedural summary once a ranking signal is available.

### Convention Index — removed from START_HERE; conventions become a first-class artifact

Conventions should not live as a field on `.aindex` files and should not be LLM-hallucinated in START_HERE. They deserve to be a proper artifact with their own lifecycle — similar to how Concepts evolved from ad-hoc mentions into a structured wiki.

See [Open Thread: Conventions](#open-thread-conventions-as-a-first-class-artifact) for the full analysis of what this entails.

## Open Thread: Conventions as a First-Class Artifact

> **Status**: Decisions settled in [plans/conventions-artifact.md](conventions-artifact.md).
> Summary retained here for context; see that document for decisions D1-D8.

Conventions are currently broken end-to-end in Lexibrary. The only place they
exist is in LLM-hallucinated START_HERE output. The `.aindex` `local_conventions`
field is hard-coded to `[]`. The link graph `conventions` table is empty. `lexi
lookup`'s "Applicable Conventions" renders nothing.

**Decided**: Conventions become a file-based artifact type, mirroring Concepts:
- Stored in `.lexibrary/conventions/` as markdown files with YAML frontmatter
- Separated from `.aindex` entirely (resolves regeneration/overwrite problem)
- Committed to git (all Lexibrary artifacts should be committed)
- Supports directory and pattern-based scopes (`*_test.py`, `src/**/*.ts`)
- Auto/manual sign-off config for LLM-extracted conventions
- `lexi lookup` displays max 5 conventions per file (configurable)
- Conflicts resolved by specificity override + validation warning

## Open Thread: Navigation by Intent

> **Status**: Expanded into [plans/navigation-by-intent.md](navigation-by-intent.md).
> Summary retained here for context; see that document for full analysis.

Navigation by Intent is the task-to-file routing table. It's the only START_HERE
section with no real alternative elsewhere in Lexibrary. Questions to investigate:

**What makes a good navigation table?**
- Maps *agent tasks* ("add a CLI command", "fix a bug in X") to *starting points* (files or directories to read first)
- Needs to reflect actual project workflows, not just structural layout
- Should be stable (doesn't change every commit) but responsive to architectural shifts

**Can it be procedurally generated?**
- The link graph has structural data: most-imported files, most-referenced concepts, highest-degree nodes. These are candidates for "important files" but not task mappings.
- Design file frontmatter descriptions provide file-level "what this does" — could an LLM map from that to "what task would lead you here"?
- Stack posts capture actual agent questions — their titles and tags reveal real workflows. Could frequent Stack topics seed the navigation table?

**Should it stay LLM-generated?**
- The editorial judgement of "what are the 5–10 most common tasks in this project?" is genuinely hard to do procedurally
- But the current LLM input is too thin (only billboards) — richer inputs could dramatically improve quality
- Candidate additional inputs: design file descriptions, concept titles/tags, Stack post titles, link graph edge counts (most-imported = most-navigated)

**Is this actually needed in START_HERE?**
- Alternative: a `lexi navigate` command that takes a task description and returns suggested files (via link graph + FTS search)
- This would be dynamic and always current, but requires the agent to know what to ask
- The static table's value is that agents don't need to formulate a query — they scan a list

**Decision needed**: Keep LLM-generated with better inputs? Make partially procedural? Replace with a `lexi navigate` command? This needs prototyping and evaluation.

## Open Thread: Navigation Protocol — Holistic Review

> **Status**: Expanded into [plans/navigation-protocol-review.md](navigation-protocol-review.md).
> Summary retained here for context; see that document for full analysis.

Navigation protocol currently appears in 6+ places:
1. `base.py` core_rules (injected into CLAUDE.md / .cursor/rules / AGENTS.md)
2. `claude.py` hooks (mechanically enforced pre/post-edit)
3. `cursor.py` `lexibrary-editing.mdc` (conditional rule)
4. `docs/agent/orientation.md` (Step 1, Step 3, "After Orientation")
5. `docs/agent/quick-reference.md` (session start checklist, before/after edit checklist)
6. START_HERE.md (LLM-regenerated each time — wording drifts)

Removing it from START_HERE eliminates one source of drift. But the remaining 5 are
not obviously coordinated. If `base.py` core_rules changes, does `orientation.md`
get updated? Does `quick-reference.md`? They're maintained by different processes
(code generation vs. manual docs).

**What needs investigation:**
- Should `orientation.md` and `quick-reference.md` be generated from `base.py` core_rules (single source of truth)?
- Or should they be distinct: rules = terse machine instructions, docs = narrative explanations?
- The hooks in `claude.py` mechanically enforce a subset of the protocol — is the overlap with text rules a feature (belt and suspenders) or a maintenance hazard?

This is not blocking for START_HERE changes but should be tracked as a related concern.

## Summary

START_HERE tries to do too much with too little input. Three of its five sections have better sources of truth elsewhere in Lexibrary. The fix is:

- **Keep**: Topology (procedural, adaptive to project scale) + Navigation by Intent (pending deeper analysis)
- **Remove**: Navigation Protocol (already in agent rules) + Convention Index (needs systemic fix, not a START_HERE patch)
- **Replace**: Ontology → pointer to `lexi concepts` (monitor, expand later)
- **Spawn**: Conventions as a first-class artifact (separate workstream, biggest effort)
